appnTransmissionGroup

The class appnTransmissionGroup defines the fields that apply to objects of APPN transmission groups. An APPN transmission group provides a route for a session stage between two logically adjacent nodes. The nodes can be any type 2.1 nodes, including:

  • APPN network nodes
  • APPN end nodes
  • APPN branch network nodes
  • Interchange nodes
  • LEN nodes
  • Migration Data Host nodes

Each APPN transmission group between two nodes is uniquely identified by a number, called the transmission group number. Using a series of transmission group numbers and names of associated adjacent nodes, a route is calculated and established for a session across an APPN network.

For the purposes of route calculations, a transmission group's state is either operational or not operational. The transmission group's state is unidirectional as seen from each of the two adjacent nodes. A transmission group must be operational in both directions to be eligible for route calculation.

The logical link to be used for the transmission group can be active or inactive for an operational transmission group. During the establishment of the route, inactive logical links are automatically activated on demand. When all the sessions using the transmission group are ended, the logical link is automatically deactivated when it uses a limited resource like a dial-up port, but the transmission group remains operational.

An APPN transmission group also provides the association between a real type 2.1 node and a virtual routing node. The virtual routing node is not a real node. Instead, it is used to name a connection network. APPN route selection uses the transmission group from each of two real nodes to the virtual routing node to establish a direct-link connection between the two nodes.

Part 4. MultiSystem Manager Data Model

Chapter 8. MultiSystem Manager Data Model Overview

This chapter is an overview of the MultiSystem Manager data models. The data models reside in RODM and are used by the NetView program to determine the status and configuration of resources managed by MultiSystem Manager. MultiSystem Manager data models are also used by the discovery manager.

MultiSystem Manager runs on a z/OS system with the NetView program. The MultiSystem Manager application operates in conjunction with the topology agents. The topology agents forward topology information to MultiSystem Manager. MultiSystem Manager stores the information in RODM, enabling the Graphic Monitor Facility host subsystem (GMFHS) to graphically display the information on the NetView management console.

The MultiSystem Manager agents are installed, started, and maintained separately from the manager function. For more information about the MultiSystem Manager agents, see IBM Tivoli NetView for z/OS Installation: Configuring Graphical Components.

This section describes the MultiSystem Manager data models. Each data model contains a hierarchical structure of classes. Each class defines a set of fields.

The MultiSystem Manager data models have many classes linked together developing parent and child relationships that form the hierarchical class structure. MultiSystem Manager, using its data model, displays objects in views, enables navigation between those views, and aggregates resource status up from real resources to aggregate resources to provide network status.

The views displayed at the NetView management console are a result of the topology data model definitions loaded in the RODM data cache and objects reported by the topology agents. The views are updated based on the agents reporting changes to the network and local topologies. Creating Links in a View

A RODM link is required to display object relationships. A RODM link is defined using a pair of fields. A field of one object points to a field on another object. MultiSystem Manager defines the links between objects that it discovers, and GMFHS creates views based on these links.

For a description of how MultiSystem Manager uses View_Information_Object class objects, see “MultiSystem Manager View Considerations” on page 195. Creating Relationships Between Views

MultiSystem Manager defines links between objects to support the following view types:

  • More detail logical
  • More detail physical
  • Configuration child
  • Configuration parent

This section describes the fields that MultiSystem Manager uses to define links between aggregate objects it creates, and the fields that you can use to define links between MultiSystem Manager-created objects and objects that you create.

More Detail View Navigation

MultiSystem Manager-created aggregate node objects are linked to child objects in the member field. Child node objects are linked to parent aggregate objects in the memberOf field. MultiSystem Manager-created aggregate link objects are linked to child objects in the memberArcs field. Child link objects are linked to parent aggregate objects in the memberArcsOf field.

You can use the following fields to define links between objects you create and MultiSystem Manager-created objects:

  • ComposedOfLogica
  • ComposedOfPhysical
  • IsPartOf

The ContainsLogical and ContainsPhysical fields on each aggregate object define which fields support more detail navigation. Use the ComposedOfLogical field for logical aggregate objects and use the ComposedOfPhysical field for physical aggregate objects. To determine if a MultiSystem Manager-created aggregate object is logical, physical or both, refer to the class description to determine if the ComposedOfLogical and ComposedOfPhysical field is defined on the class.

Objects are defined as logical, physical, or both by the ContainsLogical and ContainsPhysical OP statements in the FLCSDM4 load file. If an aggregate object is defined as both logical and physical, two views are displayed when you request a More Detail view.

Configuration Parent and Configuration Child Navigation

Configuration parent and configuration child Links are defined at the class level by the View_Information_Object_Class definitions in load file FLCSDM4.

Links are dynamically set between object classes using the ChildAccess and ParentAccess fields according to the unique configuration of a network. For classes

that inherit from the realNode class, links set on the memberOf field also define more detail and configuration parent relationships. These links are set after the MultiSystem Manager INITTOPO and GETTOPO commands are issued.

The relationships defined in the FLCSDM4 load file are added to the relationships defined in the GMFHS View_Information_Object class definitions so that you can create your own links between objects that you have created and MultiSystem Manager-created objects. Use the ChildAccess and ParentAccess fields to define the links between objects that are created by MultiSystem Manager and objects that you have defined.

If you have enabled correlation for your network, the correlation function creates the links between real resource objects and correlated aggregate objects.

MultiSystem Manager Status Aggregation

Status aggregation can be adjusted to determine how the status of a real resource affects the status of one or more aggregate objects.

See “Setting MultiSystem Manager Aggregation Thresholds and Priorities” on page 198 for more information on aggregation thresholds and priorities. For a description of the GMFHS aggregation process, refer to the IBM Tivoli NetView for z/OS Resource Object Data Manager and GMFHS Programmer's Guide.

MultiSystem Manager Status Aggregation Path

In general, resource status aggregates up to aggregate objects along the more detail navigation path. This relationship is controlled by the member and memberOf fields. For nodes, this relationship is controlled by the memberArcs and memberArcsOf fields for link objects.

The Network_View_Class is the top-level MultiSystem Manager-defined view. Network views are predefined views; each network view must be explicitly defined.
